lib/supabase/server.ts - For Server Components

import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

export async function createClient() {
  const cookieStore = await cookies()
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll() },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value, options }) =>
            cookieStore.set(name, value, options)
          )
        }
      }
    }
  )
}

lib/supabase/client.ts - For Client Components

import { createBrowserClient } from '@supabase/ssr'

4 Create Waitlist Table in Supabase

File: supabase/migrations/20260329_create_waitlist.sql

-- Create waitlist table
CREATE TABLE IF NOT EXISTS waitlist (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Add indexes
CREATE INDEX idx_waitlist_email ON waitlist(email);
CREATE INDEX idx_waitlist_created_at ON waitlist(created_at DESC);

-- Enable RLS
ALTER TABLE waitlist ENABLE ROW LEVEL SECURITY;

-- Allow anyone to insert
CREATE POLICY "Anyone can sign up for waitlist" ON waitlist
  FOR INSERT
  WITH CHECK (true);

-- Only authenticated users can view
CREATE POLICY "Authenticated users can view waitlist" ON waitlist
  FOR SELECT
  USING (auth.role() = 'authenticated');

5. Email Signup Troubleshooting

The Problem

❌ Error: "Failed to fetch" when submitting email signup form

Debugging Process

1 Initial Error Investigation

Browser console showed:

Error: TypeError: Failed to execute 'set' on 'Headers': Invalid value

This indicated an issue with HTTP headers in the Supabase client

2 Added Environment Variable Validation

Modified lib/supabase/client.ts:

export function createClient() {
  const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL
  const supabaseAnonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY

  if (!supabaseUrl || !supabaseAnonKey) {
    throw new Error(
      'Missing Supabase environment variables. ' +
      `URL: ${supabaseUrl ? 'set' : 'missing'}, ` +
      `Key: ${supabaseAnonKey ? 'set' : 'missing'}`
    )
  }

  return createBrowserClient(supabaseUrl, supabaseAnonKey)
}
3 Created Debug Endpoint

File: app/debug-env/page.tsx

This page displays all environment variables in the browser to verify they're being set correctly

4 Created API Test Endpoint

File: app/api/test-waitlist/route.ts

Server-side test that:

  • Connects to Supabase
  • Tests SELECT query
  • Tests INSERT operation
  • Tests DELETE operation
  • Returns detailed error messages
5 Root Cause Found!
The Problem: Environment variables had trailing newline characters!

https://weivmnsonkkocnsovnjw.supabase.co\n
eyJhbGc...FxFpfqw\n

These newlines were causing the HTTP Header validation to fail.
6 Why Did This Happen?

When using Vercel CLI with printf:

printf "value\nn\n" | vercel env add KEY production

The \n characters were being included in the stored value!

7 The Solution

Method 1: Use Vercel Dashboard (RECOMMENDED)

  1. Go to Vercel Project → Settings → Environment Variables
  2. Delete the corrupted variables
  3. Add them manually through the web interface
  4. Paste values carefully - no extra spaces or newlines
  5. Redeploy

Method 2: Use printf without newline

printf "value" | vercel env add KEY production

But dashboard is more reliable!

8 Verification

After fixing:

$ curl https://ucuncupersonalfinanceapp.vercel.app/api/test-waitlist

{
  "success": true,
  "message": "Waitlist table is working correctly!"
}

Email signup form now works perfectly! ✓

💡 Critical Lessons:
  • Always use Vercel Dashboard for environment variables - CLI can introduce hidden characters
  • Create debug endpoints early - They save hours of troubleshooting
  • Test server-side and client-side separately - Helps isolate issues
  • Log everything during debugging - Console logs revealed the newline issue
  • Environment variable issues are silent killers - Build succeeds but runtime fails

6. Key Lessons Learned

Development Workflow

1. Plan Before Code
  • MVP tracker kept us focused
  • Clear acceptance criteria prevented scope creep
  • Breaking into 14 tickets made progress trackable
2. Test Locally First
  • Always run npm run build before deploying
  • Create test pages for new features
  • Verify Supabase connections in development
3. Environment Variables
  • Use Vercel Dashboard, not CLI
  • Create .env.local.example for documentation
  • Verify variables are loading with debug pages
  • Check for hidden characters (newlines, spaces)
4. Debugging Strategy
  • Add extensive console logging
  • Create debug endpoints
  • Test API routes separately from UI
  • Check browser Network tab for failed requests

Technical Decisions

5. Supabase SSR Pattern
  • Separate client/server utilities prevent build errors
  • Middleware handles session refresh automatically
  • Row Level Security protects data at database level
6. Norwegian Market Focus
  • All landing page content in Norwegian
  • Targets specific banks (DNB, Nordea, Sparebank 1)
  • No need for multi-language support in MVP
7. Public vs Private Information
  • Keep technical MVP tracker private
  • Public roadmap shows features, not implementation
  • Collect emails early for launch notification

Git & Deployment

8. Commit Often with Clear Messages
  • Each logical change = one commit
  • Reference ticket numbers (T-01, T-02)
  • Describe "what" and "why" in commit messages
9. Vercel Deployment
  • Auto-deploys on git push (after setup)
  • Environment variables need manual configuration
  • Build cache speeds up subsequent deploys
  • Preview deployments help test before production

Common Pitfalls Avoided

  • ❌ Building without planning → ✅ Created MVP tracker first
  • ❌ Overengineering → ✅ Kept to MVP scope
  • ❌ No monitoring → ✅ Added Supabase dashboard access
  • ❌ Complex auth in MVP → ✅ Postponed to T-02
  • ❌ Manual deployment → ✅ Automated with Vercel
